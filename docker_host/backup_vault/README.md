# 🔐 Sovereign Vaultwarden Backups

This module handles all backup, encryption, and restoration operations for the Vaultwarden password manager running on `docker-host` (LXC 100).

**Encryption model**: AES-256 GPG symmetric encryption. The key is derived deterministically from your Bitwarden Master Password using SHA-256 + a static salt. **The key is NEVER stored on disk** — it is derived live and wiped from memory immediately after use.

---

## 📜 Script Manifest

### 1. `backup_vault_server.sh` — Automated Weekly Backup (Cron)
| | |
| :--- | :--- |
| **Deployed at** | `/root/docker_host/backup_vault/backup_vault_server.sh` on `docker-host` |
| **Schedule** | **Every Sunday at 02:00** — `0 2 * * 0` |
| **Trigger** | Cron on `docker-host` (root crontab) |
| **Prerequisites** | `/root/.vault_secret` must exist (generated by `gen_encrypt_key.sh`), `rclone` configured for `gdrive-x`, `sqlite3` installed |
| **What it does** | 1. Reads GPG key from `/root/.vault_secret`. 2. Atomically dumps `db.sqlite3` (hot backup, no downtime). 3. Copies RSA keys and config. 4. Compresses to `.tar.gz`. 5. Encrypts to `.tar.gz.gpg` (AES-256). 6. Uploads to `gdrive-x:Homelab/Backups/Vault`. 7. Applies cloud retention (keeps 3 most recent). 8. Cleans up all temp files. |
| **Retention** | Cloud: **3 most recent** |
| **Logs** | `/var/log/vault-backup.log` |
| **Notifications** | Telegram (LOCK at start, SUCCESS or ERROR at end) |

**Crontab entry on `docker-host`:**
```cron
0 2 * * 0 /root/docker_host/backup_vault/backup_vault_server.sh >> /var/log/vault-backup.log 2>&1
```

### 2. `backup_vault_local.sh` — Manual Local Backup (run from PC)
| | |
| :--- | :--- |
| **Deployed at** | Local repo only — `sh/docker_host/backup_vault/backup_vault_local.sh` |
| **Trigger** | **Manual** — run from your local PC |
| **Prerequisites** | SSH alias `docker-host` configured, HDD mounted at `/media/kpihx/KpihX-Backup`, `gpg` + `openssl` + `rclone` installed locally |
| **What it does** | 1. Asks for your **Bitwarden Master Password** (hidden, never stored). 2. Derives the GPG key from it using `openssl sha256 + salt`. 3. SSHes into `docker-host` and atomically dumps the SQLite DB and RSA keys. 4. Pulls the data to the local PC via `scp`. 5. Compresses and encrypts (AES-256) locally. 6. Copies the `.gpg` archive to the external HDD. 7. Applies HDD retention (keeps 3). 8. Uploads to Cloud and applies Cloud retention (keeps 3). 9. Wipes the derived key from memory. 10. Deletes all local temp files. |
| **Retention** | HDD: **3 most recent** / Cloud: **3 most recent** |
| **Key storage** | 🚫 NEVER stored — derived live, `unset` immediately after use |
| **Notifications** | Telegram (INFO at start, SUCCESS or ERROR at end) |

### 3. `restore_vault.sh` — Disaster Recovery
| | |
| :--- | :--- |
| **Deployed at** | `/root/docker_host/backup_vault/restore_vault.sh` on `docker-host` |
| **Trigger** | **Manual only** — disaster scenario |
| **What it does** | 1. Lists all available backups on `gdrive-x:Homelab/Backups/Vault`. 2. Asks which backup to restore (interactive). 3. Asks for **Bitwarden Master Password** to derive the decryption key. 4. Downloads the selected archive from Cloud. 5. Decrypts it in-memory. 6. Stops the Vaultwarden container. 7. Injects the restored data into the Docker volume. 8. Restarts Vaultwarden. 9. Wipes the key and all temp files. |
| **Key storage** | 🚫 NEVER stored — derived live, `unset` immediately after use |
| **Notifications** | None (interactive, run manually) |

## 🔑 Key Derivation Reference

The encryption/decryption key is **deterministically derived** from:
- Your **Bitwarden Master Password** (you provide interactively)
- A **static salt**: `Homelab-Sovereign-Vault-2026-KpihX`

Formula:
```bash
KEY = SHA-256( MasterPassword + Salt )
```

This means: **If you know your Master Password, you can always recover the key.** No key file to lose. The resulting 64-character hex key is passed directly to GPG and immediately wiped from bash variables with `unset`.

## 🛠️ `lib/gen_encrypt_key.sh` — Key Derivation Tool
| | |
| :--- | :--- |
| **Deployed at** | `/root/docker_host/lib/gen_encrypt_key.sh` on `docker-host` |
| **Trigger** | **Manual, one-time** (first setup or key re-initialisation) |
| **What it does** | Interactively asks for your Bitwarden Master Password, derives the 64-char hex key, and **displays it** so you can paste it into `/root/.vault_secret` on `docker-host`. Must be run with `chmod 400 /root/.vault_secret` afterwards to secure it. |

**Usage:**
```bash
# On docker-host
/root/docker_host/lib/gen_encrypt_key.sh
# Copy the displayed key and run:
echo "<derived_key>" > /root/.vault_secret && chmod 400 /root/.vault_secret
```

> ⚠️ The `.vault_secret` file contains the **server's copy** of the key used for automated nightly backups. The local/restore scripts derive the key on-the-fly and never store it.

## ⚙️ Setup Procedure (First Deploy or Fresh Container)

### On `docker-host`:

1. **Deploy the scripts**:
   ```bash
   scp -r docker_host/ root@docker-host:/root/docker_host/
   ssh docker-host "chmod +x /root/docker_host/backup_vault/*.sh /root/docker_host/lib/*.sh"
   ```

2. **Configure the environment file**:
   ```bash
   ssh docker-host "cp /root/docker_host/.env.example /root/docker_host/.env && nano /root/docker_host/.env"
   # Required:
   # TELEGRAM_HOMELAB_TOKEN=<bot-token>
   # CHAT_ID=<your-chat-id>
   ```

3. **Initialise the vault secret**:
   ```bash
   ssh docker-host "/root/docker_host/lib/gen_encrypt_key.sh"
   # Copy the displayed key, then:
   ssh docker-host "echo '<key>' > /root/.vault_secret && chmod 400 /root/.vault_secret"
   ```

4. **Configure rclone for root**:
   ```bash
   ssh docker-host "rclone config"  # Create remote: gdrive-x
   ```

5. **Configure crontab**:
   ```bash
   ssh docker-host "crontab -e"
   ```
   Add:
   ```cron
   0 2 * * 0 /root/docker_host/backup_vault/backup_vault_server.sh >> /var/log/vault-backup.log 2>&1
   ```

6. **Validate**:
   ```bash
   ssh docker-host "/root/docker_host/backup_vault/backup_vault_server.sh"
   ```
   Expect a Telegram notification and a new file at `gdrive-x:Homelab/Backups/Vault`.

## 🏗️ Restoration Procedure — Step by Step

Run on `docker-host` when Vaultwarden data is lost or corrupted.

```bash
ssh docker-host "/root/docker_host/backup_vault/restore_vault.sh"
```

The script will:
1. List available backups on Google Drive.
2. Ask you which one to restore.
3. Ask for your **Bitwarden Master Password** (used to derive the decryption key — never stored).
4. Download and decrypt the archive.
5. Stop Vaultwarden, inject the data, restart it.

If the Docker volume is completely gone, recreate the stack first:
```bash
ssh docker-host "cd /root && docker compose up -d vaultwarden"
# Wait for the volume to be created, then run restore_vault.sh
```
